# Аудит Premier Design v2.0 — Стратегический план развития и улучшений

Индекс: [`docs/README.md`](../README.md) · ADR: [`docs/adr/`](../adr/README.md)

Настоящий аудит сформирован с учетом современных стандартов мировой разработки (Next.js 14+, React 19, Panda CSS, headless UI-подход), требований к производительности (Core Web Vitals > 90), строгой безопасности и чистой архитектуры.

## Статус документа

- Версия: `2.0`
- Статус: `Актуален`
- Область: архитектура, UI/верстка, производительность, безопасность, инженерные практики

---

## 1. Архитектура и чистый код (Clean Architecture & FSD)
*Цель: Предсказуемость, строгие границы слоев, снижение связности.*

| ID | Задача | Описание и комментарий | Приоритет |
|----|--------|------------------------|-----------|
| **A-1** | **Изоляция Data Access Layer (DAL)** | Вынести все прямые обращения к БД (`pg-promise`) и внешним API в изолированный слой `services/dal/`. В UI и бизнес-логике использовать только абстрактные интерфейсы. Это предотвратит утечку секретов и упростит мокирование. | P1 |
| **A-2** | **Миграция на App Router (Next.js)** | Постепенный перенос страниц из `pages/` в `app/`. Это позволит использовать React Server Components (RSC), что радикально уменьшит размер клиентского бандла и ускорит First Load. | P2 |
| **A-3** | **Внедрение Server Actions** | Замена традиционных API-роутов (`pages/api/feedback`) на Server Actions с инкапсулированной валидацией (Zod) и авторизацией. Снижает boilerplate-код и улучшает типизацию между клиентом и сервером. | P2 |
| **A-4** | **Строгий Feature-Sliced Design (FSD)** | Настроить `eslint-plugin-boundaries` для жесткого контроля импортов: `features` не могут импортировать другие `features`, `shared` не зависит от бизнес-логики. | P2 |

## 2. Глубокий аудит UI/UX и Верстки (Design & Markup)
*Цель: Мировой уровень интерфейсов, идеальный a11y, fluid-дизайн и типобезопасная стилизация без Tailwind.*

| ID | Задача | Описание и комментарий | Приоритет |
|----|--------|------------------------|-----------|
| **U-2** | **Fluid Typography & Spacing** | Избавиться от жестких привязок к `vw`/`vh` (они ломают доступность при зуме в браузере). Использовать функцию `clamp()` для шрифтов и отступов, а также CSS Grid `minmax()` для адаптивных сеток без лишних `@media`. | P1 |
| **U-3** | **Z-index Scale (Слои)** | Убрать хардкод `z-index: 9999` и `1000`. Внедрить строгую шкалу токенов в `tokens.css`: `--z-hide`, `--z-base`, `--z-dropdown`, `--z-sticky`, `--z-modal`, `--z-toast`. | P2 |
| **U-4** | **Продвинутый a11y форм** | В `FeedbackForm` и `CostInput`: связать ошибки с полями через `aria-invalid` и `aria-errormessage`. Добавить `aria-live="polite"` для динамических результатов калькулятора, чтобы скринридеры озвучивали изменение цены. Убедиться в наличии атрибутов `autocomplete`. | P1 |
| **U-5** | **Полноценный Dark Mode** | В `tokens.css` сейчас описаны цвета только для `:root`. Необходимо создать блок `[data-theme="dark"]` (или `@media (prefers-color-scheme: dark)`) и переопределить семантические токены (`--color-bg`, `--color-surface`, `--color-text`) для темной темы. | P2 |
| **U-6** | **Компонентный API (Polymorphism)** | Доработать `UiButton` и другие примитивы: добавить поддержку `forwardRef` и полиморфизма (проп `as` или `asChild` как в Radix UI), чтобы кнопку можно было семантично рендерить как ссылку `<a>` без дублирования стилей. | P2 |
| **U-7** | **Интеграция Panda CSS (вместо Tailwind)** | Внедрить `@pandacss/dev` с генерацией utility/recipe-классов из токенов `styles/tokens.css`. Это дает utility-подход, но сохраняет типобезопасность и контроль дизайн-системы без длинных inline class-строк. | P2 |
| **U-8** | **Headless-компоненты вместо тяжелого UI-kit** | Для сложного интерактива использовать headless-подход (например, Radix primitives) поверх собственных токенов/recipes, а не массовую миграцию на монолитный UI-kit. | P3 |

## 3. Производительность (Performance > 90 Core Web Vitals)
*Цель: Мгновенная загрузка, плавность интерфейса, оптимизация ресурсов.*

| ID | Задача | Описание и комментарий | Приоритет |
|----|--------|------------------------|-----------|
| **P-1** | **Внедрение React Compiler** | Проект использует React 19. Включение React Compiler (через `next.config.js`) автоматически мемоизирует компоненты, избавляя от ручных `useMemo`/`useCallback` и исключая лишние рендеры. | P1 |
| **P-2** | **Агрессивный Code Splitting** | Анализ через `@next/bundle-analyzer`. Вынос тяжелых библиотек (например, `react-chatbot-kit`, иконки, сложные формы) в ленивую загрузку (`next/dynamic` или `React.lazy`). | P1 |
| **P-3** | **Оптимизация LCP и Изображений** | Перевод всех `next/image` на форматы `AVIF/WebP`. Настройка `priority` строго для LCP-элементов (первый экран). Удаление `blurDataURL` с полными картинками (уже начато, нужно завершить аудит). | P2 |
| **P-4** | **Оптимизация шрифтов и скриптов** | Использование `next/font` с `display: swap` (уже есть, проверить предзагрузку). Сторонние скрипты (аналитика, чаты) загружать через `next/script` со стратегией `worker` или `lazyOnload`. | P2 |

## 4. Безопасность (Security)
*Цель: Защита от XSS, CSRF, утечек данных и брутфорса.*

| ID | Задача | Описание и комментарий | Приоритет |
|----|--------|------------------------|-----------|
| **S-1** | **Content Security Policy (CSP)** | Внедрение строгих заголовков CSP в `next.config.js` (или через Middleware). Запрет `unsafe-inline` скриптов, настройка `nonces` для инлайн-стилей/скриптов. | P1 |
| **S-2** | **Data Tainting (React 19)** | Использование `experimental_taintUniqueValue` и `taintObjectReference` для секретов БД, токенов Telegram/SMTP, чтобы они физически не могли быть переданы в клиентские компоненты. | P1 |
| **S-3** | **Rate Limiting & DDOS Protection** | Усиление текущего `rateLimit` на всех публичных эндпоинтах (отправка форм, калькуляторы). Интеграция с Redis или in-memory хранилищем на уровне Middleware. | P2 |
| **S-4** | **Сквозная валидация (Zod)** | Строгая проверка всех входящих данных (body, query, params) на сервере + дублирование логики на клиенте (через `react-hook-form` + `zodResolver`). | P1 |

## 5. Масштабируемость, Рефакторинг и Консистентность
*Цель: Удобство поддержки, снижение когнитивной нагрузки, отсутствие логических дыр.*

| ID | Задача | Описание и комментарий | Приоритет |
|----|--------|------------------------|-----------|
| **R-1** | **Рефакторинг калькулятора (`useCalculatorHandlers`)** | Разделение сложной логики расчетов на чистые функции (pure functions), независимые от React. Покрытие их 100% unit-тестами для гарантии точности финансовых расчетов. | P2 |
| **R-2** | **Глобальный State Management** | Оценка необходимости (Zustand/Jotai). Если пропсы передаются глубже 3 уровней (`prop drilling` в Layout/PageData), перевести глобальные состояния (тема, корзина/выбор услуг) в атомарный стейт. | P3 |
| **R-3** | **Обработка ошибок (Error Boundaries)** | Внедрение глобальных и локальных `ErrorBoundary` (или `error.tsx` в App Router) для graceful degradation. Пользователь не должен видеть белый экран при падении отдельного виджета. | P2 |

## Definition of Done
1. Код проходит `yarn lint`, `yarn test`, `yarn build` без предупреждений (включая линтер React Compiler).
2. Lighthouse CI показывает >90 по всем метрикам (Performance, Accessibility, Best Practices, SEO).
3. Отсутствуют уязвимости (`npm audit` чист, CSP заголовки присутствуют в ответах сервера).
4. UI-компоненты используют токены/Panda CSS recipes и корректно работают с клавиатуры (`:focus-visible`, ARIA-атрибуты).
5. Бизнес-логика покрыта тестами (Statements > 80% для критичных модулей вроде калькулятора и форм).

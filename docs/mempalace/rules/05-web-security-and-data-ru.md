# Безопасность и данные (веб-приложение)

Практики уровня **OWASP-lite** для типичного B2C/B2B сайта с формами и API.

## Секреты и конфигурация

- Секреты только из **переменных окружения** и секрет-хранилищ CI; никогда в репозитории и клиентском бандле.
- На клиент попадают только **`NEXT_PUBLIC_*`** (или аналог), осознанно: всё публично для посетителя.

## Ввод и вывод

- Query, body, env — **валидация** (схема в духе Zod) до использования в логике.
- Экранирование при вставке пользовательских данных в **email, Telegram, PDF** и другие вторичные каналы.
- **Идемпотентность** там, где пользователь может дважды отправить форму (дубликаты заявок, платежи).

## Транспорт и заголовки

- В продакшене — **HTTPS**; заголовки безопасности (CSP, HSTS, frame-ancestors и т.д.) согласуются с хостингом и не ломают легитимные сценарии (embed, аналитика).
- Ограничение методов и размеров тела на API; **rate limit** на чувствительных эндпоинтах.

## Ошибки и логи

- Клиенту — **безопасные** сообщения; в лог — детали без PII и секретов.
- Различать ожидаемые ошибки (валидация, 404) и неожиданные (500 с оповещением команды).

## Критерии готовности

- Новые эндпоинты проходят ревью на валидацию, коды ответа и утечки данных.
- Зависимости проверяются на известные уязвимости в релизном цикле.

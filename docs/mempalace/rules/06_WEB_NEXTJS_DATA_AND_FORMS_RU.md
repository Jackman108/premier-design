# Next.js: данные, формы и серверная граница

Согласованные подходы к **загрузке данных** и **мутациям** в App Router / Pages Router (в зависимости от версии проекта).

## Источник правды

- Данные для отображения — из **явных источников**: fetch на сервере, SWR/React Query на клиенте, Server Actions — по принятой в проекте схеме.
- Не смешивать в одном компоненте три разных способа без причины; для новых фич — **один** выбранный паттерн.

## Формы и мутации

- После успешной отправки — **понятное состояние UI** (success / redirect); при ошибке — сообщение и сохранение введённого где уместно.
- Оптимистичный UI — только при откате при ошибке и понятной индикации.

## API и Server Actions

- Валидация на **границе**; бизнес-правила — в use-case, а не размазаны по handler’ам.
- Не доверять клиенту: проверка прав, квот, подписи webhook’ов — на сервере.

## Кеш и revalidate

- Использовать **`revalidatePath` / `revalidateTag` / `fetch` cache** осознанно: документировать время жизни данных и сценарии инвалидации.
- Избегать «вечного» кеша для персональных или часто меняющихся данных без стратегии сброса.

## Критерии готовности

- Сценарий «двойной сабмит» и обновление страницы после POST обработаны.
- Ошибки сети и таймауты дают пользователю **действие** (повторить, связаться), а не пустой экран.

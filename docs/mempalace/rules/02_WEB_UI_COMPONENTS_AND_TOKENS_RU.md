# UI-компоненты и дизайн-система

Практики для **визуально согласованных** интерфейсов без хаоса из inline-стилей и случайных цветов.

## Дизайн-токены

- Источник визуальных констант — **токены** (например CSS variables в проектном `tokens.css`), а не «магические» hex в каждом компоненте.
- Новые примитивы (кнопка, поле, поверхность) оформляются в **общем слое примитивов** с понятным контрактом (размеры, состояния, a11y).
- **Не вводить** новый utility-стек вроде Tailwind без отдельного архитектурного решения (ADR): дублирование системы отступов и цветов ухудшает сопровождение.

## Стилизация

- **CSS Modules** (или согласованный проектный подход) для изоляции классов; избегать глобальных селекторов без префикса/области.
- **Panda CSS / recipes** — если принято в проекте: рецепты поверх токенов, а не копипаст произвольных значений.
- Критичные для **CLS** размеры изображений и медиа — задавать явно (width/height, aspect-ratio).

## Композиция компонентов

- Составные блоки страницы собираются из **примитивов и фичевых UI-модулей**, а не из копипасты одной и той же разметки.
- Варианты кнопок/типографики — через **именованные варианты** (recipe / modifier), а не через десятки boolean props.

## Антипаттерны

- «Пока сделаю на инлайнах, потом приберём» в продакшен-коде без задачи на рефакторинг.
- Импорт тяжёлой UI-библиотеки целиком ради одного виджета без code splitting / lazy.

## Критерии готовности

- Новые цвета/отступы согласованы с токенами или добавлены в токены осознанно.
- Для нового примитива есть **story** или эквивалентная визуальная документация, если это принято в проекте.

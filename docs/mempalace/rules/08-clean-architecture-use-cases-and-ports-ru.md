# Чистая архитектура: сценарии, порты и адаптеры

Дополняет [`01-web-architecture-and-boundaries-ru.md`](01-web-architecture-and-boundaries-ru.md) акцентом на **прикладном ядре** и инверсии зависимостей.

## Сценарии (use cases)

- Один сценарий — **одна причина для изменения** (отправка заявки, расчёт оффера, публикация черновика): вход → проверки → вызов портов → результат.
- Сценарий **не** знает про HTTP-статус, заголовки ответа и формат JSON страницы: это оболочка API/UI.
- Повторное использование: общие шаги выносятся в **чистые функции** или узкий доменный сервис, а не копипаста между use-case без причины.

## Порты (абстракции наружу)

- Внешние системы (почта, Telegram, платёж, файловая система, БД) — за **интерфейсами** (функции-зависимости, тонкие типы «портов»), реализация подставляется при сборке (composition root) или в тестах — дублёр.
- **YAGNI:** не вводить интерфейс без второй реализации или тестового дублёра; иначе лишняя церемония.

## Адаптеры (инфраструктура)

- `pages/api/*`, nodemailer, `fetch` к внешнему SaaS, ORM — **адаптеры**: валидация входа, маппинг ошибок, таймауты; без дублирования бизнес-правил из use-case.
- Клиентский «тонкий» адаптер (например обёртка над `fetch`) допустим в `features/*/hooks` только если не протаскивает бизнес-логику в UI-слой.

## Граница домена и представления

- **UI** подписывается на результат сценария (успех/ошибка/загрузка), не вычисляет цены и не решает, «можно ли отправить форму» бизнес-правилами — эти правила в домене/use-case.
- **DTO** на границе API — отдельно от доменных сущностей, если иначе протекает инфраструктура в ядро.

## Критерии готовности

- Новый внешний вызов не добавлен «напрямую» из JSX без прохождения через сценарий или явный порт.
- Критичный сценарий покрыт **unit** на правила + **интеграционным** тестом с моком порта при необходимости.

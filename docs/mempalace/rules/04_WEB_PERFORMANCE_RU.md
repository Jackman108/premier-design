# Производительность веб-интерфейса

Ориентиры для **быстрого первого отображения** и стабильного **Core Web Vitals**.

## LCP (Largest Contentful Paint)

- Главный визуальный элемент первого экрана — **оптимизированное изображение** или текст без блокирующих шрифтов/скриптов.
- Для hero: приоритет загрузки (`priority` / preload там, где поддерживается стеком), корректные **размеры** и формат (AVIF/WebP по возможности).
- Удалённые источники изображений — только через **разрешённый список** конфигурации (безопасность + предсказуемость).

## JavaScript

- Анализировать бандл периодически; тяжёлые зависимости — **lazy import** или динамический импорт с `ssr: false`, если SSR не нужен.
- Не подключать тяжёлые карты/редакторы на все страницы — **разделение по маршрутам**.

## Стабильность вёрстки (CLS)

- Резервировать место под изображения, виджеты, шрифты (`size-adjust` / fallback metrics по мере поддержки стека).
- Не вставлять баннеры поверх контента без резерва высоты.

## Отзывчивость (INP / бывший FID в контексте UX)

- Длинные задачи на main thread дробить; тяжёлые вычисления — **Web Worker** или сервер.
- Дебаунс/троттлинг для scroll/resize handlers.

## Кеш и доставка

- Статика с **долгим кешем** и fingerprint в имени файла там, где это принято сборкой.
- **ISR / SSG** — для контента, допускающего задержку обновления; **SSR** — когда нужна свежесть или персонализация на сервере.

## Критерии готовности

- Перед релизом — прогон **Lighthouse mobile** (или эквивалент) в согласованной среде; регрессии LCP/CLS документируются.

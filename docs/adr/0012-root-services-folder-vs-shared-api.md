# ADR 0012: каталог `services/` в корне приложения и связь с `shared/lib`

## Статус

Принято (зафиксировано по аудиту PD-R-10, cross-repo 2026-04).

## Контекст

В корне репозитория Premier Design лежит каталог **`services/`** (`emailService`, `telegramService`, `dal/feedbackDal`, `envVar`, `fileService` и т.д.) — инфраструктурные адаптеры для API routes и use-case’ов. Параллельно часть «общих» утилит живёт в **`shared/lib/`** (rate limit, sanitize, клиенты).

В других репозиториях лендингов (febcode) похожие вещи часто именуются **`shared/api`** или группируются под единым префиксом слоя.

## Решение

1. **`services/`** оставляем как **composition root инфраструктуры** для этого приложения: явные адаптеры к nodemailer, Telegram HTTP, файловой системе, переменным окружения. Импорт из `features/*/useCases` и `pages/api/*` допустим; бизнес-правила не копируют сюда из сценариев.

2. **`shared/lib/`** — переиспользуемые вещи **без привязки к конкретному внешнему SaaS** (хелперы, ограничение запросов, SEO JSON-LD без привязки к одному виджету).

3. Перенос всего `services/` в `shared/api/*` **не делаем автоматически**: массовый рефакторинг импортов и истории git без выигрыша для пользователя. При появлении второго приложения в монорепо или выносе пакета — переоценить и выделить npm-пакет `@premium/infrastructure`.

## Последствия

- Новые интеграции по умолчанию: интерфейс/порт в use-case → реализация в `services/` или узкий модуль в `shared/lib` при отсутствии внешнего IO.
- Документ MemPalace **08** и этот ADR — точка ссылки для ревью границ.

## Проверка готовности

- Use-case не импортирует nodemailer/`fetch` напрямую; только через слой `services/` или объявленный порт.

# ADR 0014: срез `entities/` и публичный API (barrel) в FSD

## Статус

Принято (зафиксировано по §9 этап 3.4 портфельного аудита 2026-04, дополняет ADR 0012 и 0013).

## Контекст

Слой **`entities/`** содержит доменные типы и мелкие презентационные DTO для карточек, SEO, кнопок и т.д. Параллельно **ADR 0012** описывает границу **`services/`** vs **`shared/lib/`** (инфраструктурные адаптеры vs переиспользуемые утилиты), а **ADR 0013** — единый **`shared/lib`**. Не хватало явного правила: **как** наружу экспортировать сущности и **куда** класть новые доменные типы, чтобы не плодить «тихие» импорты из внутренних файлов среза.

## Решение

1. **Каталог среза:** каждая доменная область — подкаталог `entities/<name>/` с локальным публичным API (по соглашению **`index.ts`** или явные `types.ts` / `model.ts`, как уже принято в репозитории).

2. **Публичный API наружу слоя:** потребители (`features/`, `widgets/`, `pages-layer/`, `app/`) импортируют **только** из **`@entities`** или из **`entities/<name>`** через задокументированный barrel — без глубоких импортов вида `@entities/foo/internal/bar` (контроль — ESLint `no-restricted-imports` + архитектурные скрипты по мере ужесточения).

3. **Связь с ADR 0012:** `entities` **не** содержат nodemailer/Telegram/`fetch` к внешним SaaS; IO остаётся в **`services/`** или узких модулях **`shared/lib`**, согласно ADR 0012.

4. **Wildcard barrel:** если правило запрещает `export *` в публичном API — не добавлять новые wildcard-реэкспорты; расширять явными именованными экспортами.

## Последствия

- Онбординг: при новом доменном типе — сначала **`entities/`**, затем use-case в **`features/`**.
- Ревью: пересечение с ADR 0012/0013 — проверять, что не смешиваются IO и чистые DTO.
- **ADR 0012** не заменяется: это ортогональные оси («инфраструктура» vs «доменные типы»).

## Проверка готовности

- Новые импорты `@entities/...` проходят ESLint и `check:architecture`.
- В `features/*/useCases` нет прямых импортов из `services/` без порта (как в ADR 0012).

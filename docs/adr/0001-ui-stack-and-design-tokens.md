# ADR 0001: UI-стек, дизайн-токены и путь к библиотеке компонентов

## Статус

Принято (2026-04-15)

## Контекст

- Сайт уже использует **React + CSS Modules + глобальные CSS-переменные** в `styles/globals.css`.
- В исходных требованиях фигурировал **NextUI**, но в кодовой базе он **не подключён**; массовая миграция экранов дала бы высокий риск регрессий и простой поставки.
- Нужен единый контракт для новых экранов и для агентских/человеческих изменений без смешения стилей «вручную» в каждом модуле.

## Решение

1. **Базовый UI-стек (зафиксирован):** существующие **CSS Modules + токены**; новые примитивы размещаются в `components/ui/` и используют только токены (`var(--…)`), без дублирования «сырых» hex в модулях по возможности.
2. **Слой токенов:** палитра и семантические алиасы вынесены в `styles/tokens.css`, подключаются из `globals.css`. Семантика (`--color-text`, `--radius-md`, …) — предпочтительный интерфейс для новых компонентов.
3. **Storybook** — каноничный каталог визуальных контрактов для примитивов `components/ui/**/*.stories.*`.
4. **NextUI / HeroUI:** не внедряются в этом ADR. Отдельное решение (ADR-002 или spike) — только после оценки размера бандла, a11y и стоимости миграции; до этого момента **не добавлять** `@nextui-org/*` без нового ADR.

## Последствия

- Фичи и use-case слои **не импортируют** `components/**` (правило ESLint сохраняется); страницы и layout собирают UI из примитивов и доменных блоков.
- Постепенный перевод существующих блоков на токены — по мере касания файлов (incremental), без «большого взрыва».

## Проверка готовности

- `npm run storybook` / `npm run build-storybook` проходят.
- Новые UI-примитивы имеют stories и опираются на `styles/tokens.css`.

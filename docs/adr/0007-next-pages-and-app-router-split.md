# ADR 0007: гибрид `pages/` и `app/` (Next.js)

**Статус:** принято  
**Дата:** 2026-04-17

## Контекст

В репозитории сосуществуют маршруты **Pages Router** (`premier-design/pages/`) и **App Router** (`premier-design/app/`). Юридические страницы вынесены в `app/documents/*` с общей оболочкой и крошками; маркетинговый сайт и API остаются на `pages/`.

## Решение

1. **Не смешивать** один и тот же публичный URL между двумя деревьями: каждый path принадлежит либо `pages/`, либо `app/`.
2. **Новые маркетинговые лендинги** (портфолио, витринные разделы услуг) — по умолчанию в `pages/` с существующим `Layout`, `getStaticProps` и `CustomHead`, пока нет явной потребности в RSC/streaming для страницы. Предварительный расчёт сметы — в модалке, не отдельным маршрутом `pages/`.
3. **Документы и изолированные зоны** с отдельным UX (юр. текст, минимальный JS) — в `app/` при сохранении визуального паритета с основным сайтом через общие виджеты/токены.
4. **Корневой `app/layout.tsx`** остаётся обязательным для App Router; глобальные стили и провайдеры дублируют контракт с `pages/_app.tsx` там, где маршруты реально заходят в `app/`.

## Последствия

- Два entry для глобальных `<head>`-подсказок: при необходимости `preconnect`/мета для всего сайта синхронизировать `pages/_app.tsx` и при релевантности — `app/layout.tsx`.
- Миграция страницы с `pages/` на `app/` — отдельная задача: проверка SEO, `generateMetadata`, дублирования провайдеров.

## Проверка готовности

- Новый маршрут не конфликтует с `pages` и `app` одновременно.
- В чеклисте релиза указано, в каком дереве живёт URL.

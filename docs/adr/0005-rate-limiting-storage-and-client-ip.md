# ADR 0005: Rate limiting — хранилище и клиентский IP

**Статус:** принято  
**Дата:** 2026-04-17

## Контекст

- Лимиты запросов реализованы в памяти процесса (`shared/lib/rateLimit.ts`).
- Для ключа используется IP клиента; при размещении за обратным прокси часто приходит `x-forwarded-for`.

## Решение

1. **In-memory** остаётся дефолтом для одного инстанса и preview: просто, без внешних зависимостей.
2. **Несколько инстансов** (горизонтальное масштабирование): общий счётчик вынести за пределы процесса — например [Vercel KV](https://vercel.com/docs/storage/vercel-kv), Upstash Redis или аналог; контракт ключа `scope:clientId` сохраняется.
3. **IP:** модуль `shared/lib/getClientIpForRateLimit.ts`. Без явного доверия к прокси (`RATE_LIMIT_TRUST_FORWARDED_FOR` не равен `1`) используется только `socket.remoteAddress`; иначе — первый (левый) адрес из `x-forwarded-for`, как задаёт пограничный прокси (например Vercel).

## Последствия

- На production за доверенным прокси нужно выставить `RATE_LIMIT_TRUST_FORWARDED_FOR=1`, иначе лимит может считаться по IP прокси.
- При нескольких репликах без общего store лимит «на пользователя» не глобален.

## Проверка готовности

- Переменная описана в `README.md` и `.env.example`.
- Поведение покрыто unit-тестами `getClientIpForRateLimit`.
